LINQ : Language Integrated Query

Définition Simple

LINQ (Language Integrated Query) est une technologie qui vous permet d'écrire des requêtes directement en C# pour rechercher, filtrer, trier et manipuler vos données.

Concrètement :
  • Au lieu d'écrire du SQL → Vous écrivez du C# naturel
  • Au lieu de boucles complexes → Vous décrivez ce que vous voulez
  • Au lieu de code verbeux → Vous écrivez des requêtes élégantes

Entity Framework + LINQ = Magie ! Vos requêtes LINQ sont automatiquement transformées en SQL optimisé.

LINQ vs SQL : Comparaison Concrète

Même requête : SQL vs LINQ

Objectif : Trouver tous les produits actifs de la catégorie "Électronique" avec un prix < 1000€, triés par prix croissant

❌ Version SQL (complexe)
SELECT p.Id, p.Name, p.Price, c.CategoryName
FROM Products p
INNER JOIN Categories c ON p.CategoryId = c.Id
WHERE c.CategoryName = 'Électronique'
  AND p.IsActive = 1
  AND p.Price < 1000
ORDER BY p.Price ASC;
✅ Version LINQ (naturelle)
var produits = context.Products
    .Where(p => p.Category.Name == "Électronique")
    .Where(p => p.IsActive)
    .Where(p => p.Price < 1000)
    .OrderBy(p => p.Price)
    .ToList();
Avantages de LINQ :
  • Lisible → Se lit comme une phrase en français
  • IntelliSense → Autocomplétion et vérification d'erreurs
  • Type-safe → Erreurs détectées à la compilation
  • Réutilisable → Fonctionne avec tous types de données

Les Deux Syntaxes LINQ

Syntaxe Query (SQL-like)
var produits = from p in context.Products
               where p.Price < 1000
               where p.IsActive
               orderby p.Name
               select p;

Ressemble à SQL → Familier si vous connaissez SQL

Syntaxe Method (Chaînage)
var produits = context.Products
    .Where(p => p.Price < 1000)
    .Where(p => p.IsActive)
                    .OrderBy(p => p.Name);

Plus puissante → Recommandée pour Entity Framework

Quelle syntaxe choisir ?

Pour Entity Framework, utilisez la syntaxe Method (chaînage) ! Elle est plus flexible, plus puissante, et c'est la norme dans la communauté .NET. C'est ce que nous utiliserons dans ce cours.

Notre Plan d'Apprentissage LINQ

Progression d'apprentissage :
  1. Requêtes de base → Where, OrderBy, Select, Take
  2. Projections et transformations → Select avec objets personnalisés
  3. Jointures et relations → Include, ThenInclude, Join
  4. Agrégations → Count, Sum, Average, Min, Max
  5. Groupements → GroupBy avec statistiques
  6. Requêtes avancées → Any, All, Contains, First vs Single
  7. Optimisations → AsNoTracking, requêtes asynchrones
  8. Débogage → Voir le SQL généré, comprendre les performances

Requêtes de Base : Vos Premiers Pas avec LINQ

Préparation : Notre Modèle d'Exemple

Pour ce module, nous utiliserons un modèle simple de boutique en ligne :

// Nos modèles pour les exemples
public class Category
{
    public int Id { get; set; }
    public string Name { get; set; }
    public ICollection<Product> Products { get; set; }
}

public class Product
{
    public int Id { get; set; }
    public string Name { get; set; }
    public decimal Price { get; set; }
    public int Stock { get; set; }
    public bool IsActive { get; set; }

1. Where : Filtrer les Données

Filtrage Simple
using var context = new ShopContext();

// ✅ Produits actifs seulement
var produitsActifs = context.Products
    .Where(p => p.IsActive)
    .ToList();

// ✅ Produits moins de 100€
var produitsPasChers = context.Products
    .Where(p => p.Price < 100)
    .ToList();

// ✅ Produits avec stock disponible
var produitsEnStock = context.Products
    .Where(p => p.Stock > 0)
    .ToList();

// ✅ Recherche par nom (contient "iPhone")
var iphones = context.Products
    .Where(p => p.Name.Contains("iPhone"))
    .ToList();
Explication détaillée :
  • p => p.IsActive → "p" représente chaque Product, on vérifie si IsActive est true
  • p => p.Price < 100 → Pour chaque produit, on vérifie si le prix est inférieur à 100
  • .Contains("iPhone") → Équivalent SQL : LIKE '%iPhone%'
  • .ToList() → Exécute la requête et récupère les résultats
Filtrage Multiple (ET logique)
// ✅ Méthode 1 : Plusieurs Where (recommandée - plus lisible)
var produits = context.Products
    .Where(p => p.IsActive)           // ET produit actif
    .Where(p => p.Price >= 50)        // ET prix >= 50€
    .Where(p => p.Price <= 500)       // ET prix <= 500€
    .Where(p => p.Stock > 0)          // ET en stock
    .ToList();

// ✅ Méthode 2 : Conditions combinées avec &&
var produits2 = context.Products
    .Where(p => p.IsActive && 
                p.Price >= 50 && 
                p.Price <= 500 && 
                p.Stock > 0)
    .ToList();

// ✅ Méthode 3 : Utiliser des variables pour plus de clarté
decimal prixMin = 50;
decimal prixMax = 500;
var produits3 = context.Products
    .Where(p => p.IsActive)
    .Where(p => p.Price >= prixMin && p.Price <= prixMax)
    .Where(p => p.Stock > 0)
    .ToList();
💡 Conseil Performance : Les multiples .Where() sont combinés en une seule clause WHERE dans le SQL final. C'est plus lisible et tout aussi performant !

2. OrderBy : Trier les Résultats

Tri Simple
// ✅ Trier par prix croissant (du moins cher au plus cher)
var parPrixCroissant = context.Products
    .OrderBy(p => p.Price)
    .ToList();

// ✅ Trier par prix décroissant (du plus cher au moins cher)
var parPrixDecroissant = context.Products
    .OrderByDescending(p => p.Price)
    .ToList();

// ✅ Trier par nom alphabétique
var parNom = context.Products
    .OrderBy(p => p.Name)
    .ToList();

// ✅ Trier par date de création (plus récents en premier)
var parDateRecente = context.Products
    .OrderByDescending(p => p.CreatedAt)
    .ToList();
Tri Multiple
// ✅ Trier par catégorie, puis par prix croissant
var triMultiple = context.Products
    .OrderBy(p => p.CategoryId)      // D'abord par catégorie
    .ThenBy(p => p.Price)            // Puis par prix croissant
    .ToList();

// ✅ Trier par stock décroissant, puis par nom alphabétique
var triComplexe = context.Products
    .OrderByDescending(p => p.Stock) // D'abord par stock (plus de stock en premier)
    .ThenBy(p => p.Name)             // Puis par nom alphabétique
    .ToList();

// ✅ Exemple réaliste : produits actifs, par catégorie, puis par prix
var produitsOrganises = context.Products
    .Where(p => p.IsActive)          // Seulement les actifs
    .OrderBy(p => p.CategoryId)      // Grouper par catégorie
    .ThenBy(p => p.Price)            // Moins chers en premier dans chaque catégorie
    .ToList();
Ordre des méthodes de tri :
  • OrderBy / OrderByDescending → Premier critère de tri
  • ThenBy / ThenByDescending → Critères suivants
  • ⚠️ N'utilisez jamais deux OrderBy → le second écrase le premier !

3. Take et Skip : Limiter les Résultats

Pagination et Limitation
// ✅ Les 10 produits les moins chers
var top10PasCher = context.Products
    .OrderBy(p => p.Price)
    .Take(10)
    .ToList();

// ✅ Les 5 produits les plus chers
var top5Cher = context.Products
    .OrderByDescending(p => p.Price)
    .Take(5)
    .ToList();

// ✅ Pagination : page 2, 10 éléments par page (ignorer les 10 premiers)
int page = 2;
int elementsParPage = 10;
var page2 = context.Products
    .OrderBy(p => p.Name)
    .Skip((page - 1) * elementsParPage)  // Ignorer les 10 premiers
    .Take(elementsParPage)               // Prendre les 10 suivants
    .ToList();

// ✅ Les 3 derniers produits ajoutés
var derniersProduits = context.Products
    .OrderByDescending(p => p.CreatedAt)
    .Take(3)
    .ToList();
Calcul de Pagination :
  • Page 1 → Skip(0), Take(10) → éléments 1-10
  • Page 2 → Skip(10), Take(10) → éléments 11-20
  • Page 3 → Skip(20), Take(10) → éléments 21-30
  • Formule → Skip((page - 1) × taille), Take(taille)

4. Select : Transformer les Données

Projections Simples
// ✅ Récupérer seulement les noms des produits
var nomsDesProduits = context.Products
    .Select(p => p.Name)
    .ToList(); // List<string>

// ✅ Récupérer seulement les prix
var prix = context.Products
    .Select(p => p.Price)
    .ToList(); // List<decimal>

// ✅ Créer des objets personnalisés
var produitsSimplifies = context.Products
    .Select(p => new 
    {
        Nom = p.Name,
        Prix = p.Price,
        EnStock = p.Stock > 0
    })
    .ToList();

// ✅ Formatage des données
var produitsAffiches = context.Products
    .Select(p => new 
    {
        NomComplet = $"{p.Name} ({p.Category.Name})",
        PrixFormatte = $"{p.Price:C}",
        StatutStock = p.Stock > 0 ? "En stock" : "Rupture"
    })
    .ToList();
💡 Optimisation : Select ne récupère que les colonnes nécessaires de la base ! C'est plus rapide que récupérer des objets complets.

5. Combiner Toutes les Opérations

Exemple Réaliste Complet
// 🎯 Objectif : Top 10 des produits électroniques actifs, 
//     moins de 500€, en stock, triés par prix, avec infos formatées

var topProduits = context.Products
    .Where(p => p.Category.Name == "Électronique")  // Catégorie spécifique
    .Where(p => p.IsActive)                         // Produits actifs
    .Where(p => p.Price < 500)                      // Budget limité
    .Where(p => p.Stock > 0)                        // En stock
    .OrderBy(p => p.Price)                          // Du moins cher au plus cher
    .Take(10)                                       // Top 10
    .Select(p => new                                // Formater pour affichage
    {
        Id = p.Id,
        Nom = p.Name,
        Prix = $"{p.Price:C}",
        Stock = $"{p.Stock} unités",
        Categorie = p.Category.Name
    })
    .ToList();

// 📊 Utilisation des résultats
foreach (var produit in topProduits)
{
    Console.WriteLine($"{produit.Nom} - {produit.Prix} ({produit.Stock})");
}
Ordre d'Exécution Optimal :
  1. Where → Filtrer d'abord (moins de données à traiter)
  2. OrderBy → Trier les données filtrées
  3. Take/Skip → Limiter après tri
  4. Select → Transformer en dernier (seulement les données finales)

Jointures et Relations : Naviguer Entre les Tables

Comprendre les Relations avec Entity Framework

Rappel Important

Entity Framework gère automatiquement les relations entre vos entités grâce aux propriétés de navigation. Vous n'avez pas besoin d'écrire des JOIN SQL complexes !

Mais attention : Par défaut, EF ne charge pas automatiquement les données liées pour éviter les problèmes de performance. Vous devez le demander explicitement.

1. Include : Charger les Données Liées

Chargement Explicite avec Include
// ❌ PROBLÈME : Sans Include, Category sera null
var produitsSansCategorie = context.Products
    .Where(p => p.IsActive)
    .ToList();
// produitsSansCategorie[0].Category == null ⚠️

// ✅ SOLUTION : Avec Include, Category est chargée
var produitsAvecCategorie = context.Products
    .Include(p => p.Category)  // Charger la catégorie liée
    .Where(p => p.IsActive)
    .ToList();
// produitsAvecCategorie[0].Category.Name est accessible ✅

// ✅ Utilisation pratique avec filtrage
var produitsElectronique = context.Products
    .Include(p => p.Category)
    .Where(p => p.Category.Name == "Électronique")
    .Where(p => p.IsActive)
    .ToList();

// ✅ Avec tri et projection
var produitsAffiches = context.Products
    .Include(p => p.Category)
    .Where(p => p.IsActive)
    .OrderBy(p => p.Category.Name)
    .ThenBy(p => p.Price)
    .Select(p => new 
    {
        Nom = p.Name,
        Prix = p.Price,
        Categorie = p.Category.Name  // Accessible grâce à Include
    })
    .ToList();
⚠️ Performance : Include génère un LEFT JOIN en SQL. N'incluez que les relations dont vous avez vraiment besoin !

2. ThenInclude : Relations Imbriquées

Charger des Relations de Relations

Imaginons un modèle plus complexe avec Order → OrderItems → Product → Category :

// Modèles étendus pour l'exemple
public class Order
{
    public int Id { get; set; }
    public DateTime OrderDate { get; set; }
    public int CustomerId { get; set; }
    public ICollection<OrderItem> OrderItems { get; set; }
}

public class OrderItem  
{
    public int Id { get; set; }
    public int OrderId { get; set; }
    public int ProductId { get; set; }
    public int Quantity { get; set; }
    public Order Order { get; set; }
    public Product Product { get; set; }
}

// ✅ Charger Order → OrderItems → Product → Category
var commandesCompletes = context.Orders
    .Include(o => o.OrderItems)              // Charger les items de commande
        .ThenInclude(oi => oi.Product)       // Puis les produits de chaque item
        .ThenInclude(p => p.Category)        // Puis la catégorie de chaque produit
    .Where(o => o.OrderDate >= DateTime.Today.AddDays(-30))
    .ToList();

// ✅ Utilisation des données chargées
foreach (var commande in commandesCompletes)
{
    Console.WriteLine($"Commande #{commande.Id} du {commande.OrderDate:dd/MM/yyyy}");
    
    foreach (var item in commande.OrderItems)
    {
        Console.WriteLine($"  - {item.Quantity}x {item.Product.Name} " +
                         $"({item.Product.Category.Name}) - {item.Product.Price:C}");
    }
}
Attention aux Performances !
  • ThenInclude peut générer des requêtes SQL très complexes
  • Cartesian explosion → 1 commande avec 10 items = 10 lignes SQL retournées
  • Alternative → Utilisez plusieurs requêtes simples si nécessaire

3. Filtrage sur Relations

Where avec Navigation Properties
// ✅ Produits d'une catégorie spécifique
var produitsElectronique = context.Products
    .Include(p => p.Category)
    .Where(p => p.Category.Name == "Électronique")
    .ToList();

// ✅ Catégories qui ont au moins un produit actif
var categoriesAvecProduitsActifs = context.Categories
    .Where(c => c.Products.Any(p => p.IsActive))
    .ToList();

// ✅ Catégories avec produits dans une gamme de prix
var categoriesPrixMoyen = context.Categories
    .Where(c => c.Products.Any(p => p.Price >= 100 && p.Price <= 500))
    .Include(c => c.Products.Where(p => p.Price >= 100 && p.Price <= 500))
    .ToList();

// ✅ Produits des catégories qui contiennent "Tech" dans le nom
var produitsTech = context.Products
    .Include(p => p.Category)
    .Where(p => p.Category.Name.Contains("Tech"))
    .OrderBy(p => p.Category.Name)
    .ThenBy(p => p.Name)
    .ToList();

4. Join Explicite (Avancé)

Join Manuel (rarement nécessaire avec EF)
// ✅ Join explicite (équivalent à Include mais plus de contrôle)
var produitsAvecCategorie = context.Products
    .Join(
        context.Categories,           // Table à joindre
        product => product.CategoryId, // Clé de Product
        category => category.Id,      // Clé de Category
        (product, category) => new    // Résultat de la jointure
        {
            ProductName = product.Name,
            ProductPrice = product.Price,
            CategoryName = category.Name
        })
    .Where(result => result.ProductPrice < 100)
    .OrderBy(result => result.CategoryName)
    .ToList();

// ✅ Left Join avec GroupJoin (pour inclure catégories sans produits)
var categoriesEtProduits = context.Categories
    .GroupJoin(
        context.Products,
        category => category.Id,
        product => product.CategoryId,
        (category, products) => new
        {
            CategoryName = category.Name,
            ProductCount = products.Count(),
            Products = products.Where(p => p.IsActive).ToList()
        })
    .ToList();
💡 Conseil : Avec Entity Framework, utilisez Include plutôt que Join explicite. C'est plus simple et EF optimise automatiquement !
